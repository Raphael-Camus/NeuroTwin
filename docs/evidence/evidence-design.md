# Evidence Design

I use evidence cards to keep the Design stage grounded. A literature note only helps NeuroTwin if it changes a scenario, a model requirement, a validation gate, or a next action.

## Card Fields

Each evidence card should answer:

- what claim did I extract?
- what design input does it affect?
- what surrogate requirement does it imply?
- what validation gate should it update?
- what risk should I track?
- what should I do next?

## Current Evidence Backbone

The current project direction comes from three lines of work:

- surrogate brain and virtual perturbation methods, especially the idea of perturbing a trained model to estimate directed response;
- digital twin brain work that links connectomes, task response, and personalized simulation;
- AI4S workflow systems that emphasize traceable loops, validation gates, and reusable scientific capabilities.

I keep the full papers local under `references/papers/` and ignore them in Git. Public-facing documentation should summarize the design implication, not redistribute the source material.
